# Proteomics Term 标识符设计规范（B64 Term Scheme, v0.1）

> 目标：在**常规蛋白 + 修饰蛋白**两种数据结构下，为所有实体提供一个**统一、可逆、跨语言、可 join 的技术主键 `Term`**。

------

## 1. 设计背景

当前蛋白/修饰组学流程中，存在两类实体：

- **常规蛋白（protein-level）**
    每行对应一个 `Accession`
- **修饰蛋白（PTM site-level）**
    每行对应 `(Accession, PtmSite)` 这一对

常见的做法是：

- 常规蛋白：join 用 `Accession`
- 修饰蛋白：join 用 `(Accession, PtmSite)`

这带来几个问题：

1. **脚本中遍地 if 分支**
    - “如果是 PTM 就用两个键，否则用一个键”，
    - 每个脚本都要判断一次，心智负担和出错点都很高。
2. **字符串拼接方案不稳定**
    - `Term = Accession_PtmSite`
    - 依赖分隔符（如_）；
    - 上游 `Accession` 本身可能包含 `_`，容易在某些物种/数据库上误切。
3. **多语言、多格式传递复杂**
    - R / Python / parquet / TSV / Excel / WDL 等联动时，若没有一个“统一、可逆”的字符主键（例如，构造成[Accession, PtmSite]），很容易出现半路丢结构信息、难以溯源的情况。

为此，我们设计一个统一方案：

> 引入 `Term` 作为技术主键， 通过标准 base64 编码 `(Accession, PtmSite)`，保证可逆、不冲突、跨语言兼容。

------

## 2. 需求与非需求

### 2.1 必须满足的需求（Requirements）

1. **统一主键：Term 唯一标识实体**

- 无论是常规蛋白还是修饰蛋白，
    所有 join / merge / 聚合都只用 `Term` 字段作为主键。

1. **可逆性：Term → (Accession, PtmSite)**

- 在只有 Term 的上下文中，必须能无损恢复 `Accession` 和 `PtmSite`；
- 允许中间步骤只保留 Term，后续再恢复结构字段。

1. **避免与外部 ID 冲突**

- Term 必须有显式前缀，区分于：
    - 上游原始 `Accession`
    - 用户自定义列名
    - 旧版 `Term` 等

1. **跨语言 / 跨格式友好**

- R / Python 实现简单、对齐；
- 可安全写入 TSV / Excel / parquet / JSON；
- 不依赖任何数据库/物种对 ID 命名的“潜规则”。

------

### 2.2 明确不追求的东西（Non-goals）

1. **Term 不需要人类可读**

- Term 是**技术 ID**，不是“给客户看的蛋白名”；
- 面向人类的展示由：
    - `Accession`
    - `PtmSite`
    - （可选）`TermLabel = Accession` 或 `Accession_PtmSite`
        负责。

1. **Term 不独自承载所有语义**

- 物种、平台、修饰类别等语义，不塞进 Term；
- 这些信息应由专门字段承担（`Species`、`PtmCategory` 等）。

1. **不在 base64 内部玩正则/模式识别**

- 不试图通过“看起来像 S123”这类 pattern 来判断 PtmSite；
- 一切结构信息只由编码/解码协议负责，不由“目测”负责。

1. **暂不将 URL-safe 当成核心要求**

- 当前 Term 主要存在于内存/文件中，不直接出现在 URL / 文件名中；
- URL-safe 可以在未来某些边界（如 API / Web 前端）通过**适配层**单独实现，而不污染核心 ID 协议。

------

## 3. 设计决策概览

### 3.1 核心思路

1. **Term 只做一件事：唯一且可逆的技术主键**
2. **语义字段（Accession / PtmSite）始终显式存在于表中**
3. **使用标准 base64 串联这两个字段，形成 Term**

> 也就是：
> **Term = 一段人类不可读的防撞字符串，但确实是 `(Accession, PtmSite)` 的可逆编码。**

------

### 3.2 不采用的方案（及原因）

1. **`Term = Accession_PtmSite` + 从右/左拆分**

- 优点：可读、简单；
- 问题：
    - Accession 本身可能包含 `_`；
    - 任何依赖字符分隔的拆分方法都在根本上不安全（理论上永远可以构造反例）。

1. **纯 hash TermID（不可逆）**

- 优点：
    - 逻辑上最干净：`TermID = hash(Accession, PtmSite)` 唯一、稳定；
- 问题：
    - 无法在“只有 Term 的表”中恢复 Accession / PtmSite；
    - 与当前“Term 需要可回溯”的需求不完全契合。

1. **JSON / TLV / 自定义 key=value 协议 + base64**

- 结构上更自描述，但对于当前仅有 `(Accession, PtmSite)` 两个字段的 schema 来说过重；
- base64 之前再套一层 key=value，不提供实质性安全提升，只增加复杂度。

综上，在“**Term 必须可逆**”的前提下，选用：

> **标准 base64（带 `+/=`）编码的双段 Term 方案。**

------

## 4. Term 编码协议（Term Encoding Protocol）

### 4.1 字段定义

在统一的数据流处理表中，包含：

- `Term`
    - **统一主键 / join 键**；
    - 内部格式为 `B64:` 开头的字符串；

在统一的正式结果展示表中，包含：

- `Accession`
    - 原始蛋白 ID
- `PtmSite`（只在修饰项目中有）
    - 修饰位点文本（例如 `S123`、`S123;Y45`）

------

### 4.2 编码规则： (Accession, PtmSite) → Term

1. 视 `Accession`、`PtmSite` 为 UTF-8 字符串；
2. 使用**标准 base64**编码：
    - 字符集：`A–Z a–z 0–9 + / =`
    - 不做任何 `url-safe` 替换；
    - 不去掉 `=` padding。

记 base64 后的结果为：

- `AccessionB64 = base64(Accession)`
- `PtmSitB64 = base64(PtmSite)`，其中：
    - `PtmSite` 为 `NA` / 空字符串时，统一编码为 `""`（空）。

1. Term 构造：

- **无 PtmSite（常规蛋白）：**

    ```text
    Term = "B64:" + AccessionB64 
    ```

- **有 PtmSite（修饰蛋白）：**

    ```text
    Term = "B64:" + AccessionB64  + "." + PtmSitB64
    ```

1. 性质：

- 标准 base64 不包含 `.`，因此 `.` 作为分隔符是天然安全的；
- `B64:` 前缀保证 Term 不与原始 ID / 旧格式 Term 混淆；
- 任意字符的 Accession / PtmSite（包含 `_`、`-` 等）都是先被完整包裹进 base64 后再拼装，避免所有“分隔符撞车”问题。

------

### 4.3 解码规则：Term → (Accession, PtmSite)

给定字符串 `Term`：

1. **检查前缀**

- 必须以 `"B64:"` 开头；
- 若否，视为**旧格式或外部数据**，解析函数应报错或交给兼容逻辑处理。

1. **去掉前缀**

```text
body = Term 去掉前 4 个字符（"B64:"）
```

1. **分割 base64 段**

- 若 `body` 中**不包含 `.`**：
    - `acc_part = body`
    - 无 `site_part`
- 若 `body` 中**包含 `.`**：
    - `acc_part = body` 的**第一个 `.` 左侧**；
    - `site_part = body` 的**第一个 `.` 右侧全部内容**。

1. **标准 base64 解码**

- `Accession = base64decode(acc_part)`
- `PtmSiteString = base64decode(site_part)`
- 若 `PtmSiteString == ""`（空），则视为 `PtmSite = NA`。

## 5. 参考实现

### 5.1 R 版本（伪代码，可直接落地）

依赖：`base64enc`

```r
library(base64enc)

encode_b64 <- function(x) {
    x <- as.character(x)
    x[is.na(x)] <- ""
    vapply(
        x,
        FUN.VALUE = character(1),
        function(s) base64enc::base64encode(charToRaw(s))
    )
}

decode_b64 <- function(s) {
    s <- as.character(s)
    vapply(
        s,
        FUN.VALUE = character(1),
        function(b) {
            if (b == "") return("")
            rawToChar(base64enc::base64decode(b))
        }
    )
}

make_term <- function(accession, site = NA_character_) {
    AccessionB64   <- encode_b64(accession)
    PtmSitB64 <- encode_b64(site)

    has_site <- PtmSitB64 != ""

    ifelse(
        has_site,
        paste0("B64:", AccessionB64 , ".", PtmSitB64),
        paste0("B64:", AccessionB64 )
    )
}

parse_term <- function(term) {
    term <- as.character(term)

    if (any(!startsWith(term, "B64:"))) {
        stop("parse_term(): Term without 'B64:' prefix. Possibly old/external format.")
    }

    body    <- substring(term, 5L)
    has_dot <- grepl("\\.", body)

    acc_part <- ifelse(
        has_dot,
        sub("\\..*$", "", body),
        body
    )
    site_part <- ifelse(
        has_dot,
        sub("^.*?\\.", "", body),
        ""
    )

    accession <- decode_b64(acc_part)
    site_str  <- decode_b64(site_part)
    site      <- ifelse(site_str == "", NA_character_, site_str)

    list(
        Accession = accession,
        PtmSite      = site
    )
}
```

### 5.2 Python 版本

依赖：标准库 `base64`

```python
import base64
from typing import Optional, Tuple

def encode_b64(s: Optional[str]) -> str:
    if s is None:
        s = ""
    s_bytes = s.encode("utf-8")
    return base64.b64encode(s_bytes).decode("ascii")

def decode_b64(b: str) -> str:
    if b == "":
        return ""
    return base64.b64decode(b.encode("ascii")).decode("utf-8")

def make_term(accession: str, site: Optional[str] = None) -> str:
    AccessionB64  = encode_b64(accession)
    PtmSitB64 = encode_b64(site)
    if PtmSitB64 == "":
        return f"B64:{AccessionB64 }"
    else:
        return f"B64:{AccessionB64 }.{PtmSitB64}"

def parse_term(term: str) -> Tuple[str, Optional[str]]:
    if not term.startswith("B64:"):
        raise ValueError(f"parse_term(): invalid Term without 'B64:' prefix: {term}")
    body = term[4:]
    if "." in body:
        acc_part, site_part = body.split(".", 1)
    else:
        acc_part, site_part = body, ""
    accession = decode_b64(acc_part)
    site_str  = decode_b64(site_part)
    site      = site_str if site_str != "" else None
    return accession, site
```

## 6. 使用规范

### 6.1 上游规范化入口

所有定量表读入后，建议统一走一层“规范化”：

```r
df_quant <- fread(c_file_quant, sep = "\t") |>
    mutate(
        Accession = as.character(Accession),
        PtmSite      = if ("PtmSite" %in% names(.)) as.character(PtmSite) else NA_character_,
        Term      = make_term(Accession, PtmSite)
    ) |>
    select(Term, Accession, PtmSite, everything())
```

> 之后所有内部表都保证：
>
> - `Term` 存在且唯一标识一行实体；
> - `Accession` / `PtmSite` 为显式结构列。

------

### 6.2 Join / 合并 / 聚合

- **一律使用 `Term` 作为 join 键：**

    ```r
    dt1[dt2, on = "Term"]
    merge(df1, df2, by = "Term")
    ```

- 下游代码**禁止**自己对 `Term` 进行字符串 `strsplit()` / 正则解析；
    如需获取 `Accession` / `PtmSite`，必须通过统一的 `parse_term()` 接口。

------

### 6.3 对外交付

- 面向客户 / 报告的交付表：
    - 必须包含：
        - `Accession`
        - `PtmSite`（修饰项目）
    - 可选包含：
        - `TermLabel`（如 `Accession` 或 `Accession_PtmSite`）
    - `Term` 列可保留用于内部溯源，也可隐藏，仅用于内部流程。

------

## 7. 迁移与兼容

1. **已有 Accession + PtmSite 的表**

    - 忽略旧 `Term` 列（若存在），或重命名为 `TermOld`；
    - 用 `make_term(Accession, PtmSite)` 生成新 Term 列；
    - 后续所有新表/新脚本统一使用新 Term。

2. **只有旧 Term（如 `Accession_PtmSite`）的表**

    - 可以视需要写一个**一次性兼容解析函数**，按旧规则 best-effort 拆分；
    - 长期建议是：从上游重新导出结构完整的 `Accession` / `PtmSite`，再生成 B64 Term。

3. **解析时遇到非 `B64:` 前缀 Term**

    - `parse_term()` 应直接抛出错误或发出清晰警告，避免悄悄误解析：

        ```text
        parse_term(): Term without 'B64:' prefix. Possibly old format or external data.
        ```

------

## 8. URL / 文件名等扩展场景（可选）

当前规范只定义 **canonical Term 表示**，不强制 url-safe。

如未来需要在 URL / 文件名中携带 Term，可通过**单独适配层**实现：

- `TermURL = term_to_url_safe(Term)`
- `Term = url_safe_to_term(TermURL)`

这类适配不改变 Term 的 canonical 定义，仅作为运输层编码。
 （例如可使用 base64url 或简单的 URL 编码。）
