## Notation

This document uses the following mathematical and structural notation:

### Sets and Number Systems

- $\mathbb{R}$ — the set of real numbers
- $\mathbb{R}^n$ — an $n$-dimensional real vector
- $\mathbb{R}^{n \times m}$ — an $n \times m$ real-valued matrix

### Matrix and Indices

- $M \in \mathbb{R}^{n \times m}$ — the numerical matrix to be encoded
- $M_{ij}$ — the value of row $i$, column $j$
- $n$ — number of rows
- $m$ — number of columns
- $R = \{r_1, r_2, \dots, r_n\}$ — set of row identifiers
- $C = \{c_1, c_2, \dots, c_m\}$ — set of column identifiers

### Ordering and Clustering

- $P$ — row permutation (ordering)
- $Q$ — column permutation (ordering)
- $P(i)$ — the position of row $i$ after ordering
- $Q(j)$ — the position of column $j$ after ordering
- $\mathcal{D}$ — distance (dissimilarity) matrix
- $\mathcal{H}$ — hierarchical clustering tree (dendrogram)

Examples of distances:

- $d_{\text{euclid}}(x,y)$ — Euclidean distance
- $d_{\text{corr}}(x,y) = 1 - \text{corr}(x,y)$

### Encoding and Color Mapping

- $f: \mathbb{R} \rightarrow \text{Color}$ — continuous color mapping function
- $\mathcal{C}$ — the color scale (sequential/diverging)
- $v_{\min}, v_{\max}$ — extremes of the value distribution
- $v_0$ — midpoint for diverging maps

### Annotations

- $A(r_i)$ — metadata/attributes associated with row $r_i$
- $B(c_j)$ — metadata/attributes associated with column $c_j$
- $\mathcal{A}_R, \mathcal{A}_C$ — row and column annotation tables

### Heatmap Rendering Components

- $\text{tile}(i,j)$ — rendered cell at $(i,j)$
- $\mathcal{L}$ — layout configuration (panels, spacing, alignment)
- $\mathcal{R}$ — rendering backend (ggplot2, d3.js, WebGL, raster engine)

## Problem Definition

> Why do heatmaps exist? What fundamental problem do they solve?

From first principles, a heatmap solves the core cognitive and computational problem: Mapping a high-dimensional continuous dataset into a 2D perceptual representation with minimal loss of structural information.

### Key Challenges

- Humans cannot read millions of numbers. Visual cortex excels at detecting **patterns, gradients, clusters, and boundaries**.
- Screens are inherently **two-dimensional**, while data often are not.
    - **Ordering** matters more than absolute magnitude.
    - **Structural similarity** is more cognitively meaningful than numeric precision.
- Thus the heatmap is *not* a stylistic choice — it is the inevitable convergence of **human perception**, **computational cost**, and **information compression**.

### Conclusion

Any visual system designed to compress high-dimensional data into 2D while preserving structure must implement the three core mechanisms of a heatmap: **coordinates** $(i, j)$, **color encoding** $f(v)$, and **sequential ordering** $P/Q$.

## Atomic Decomposition

> The irreducible building blocks of any heatmap system.

All heatmap systems, regardless of platform or domain, can be decomposed into **five indivisible “atoms”**:

### Position Atom (i, j)

The two-dimensional coordinate of a cell in the grid. This defines the spatial canvas onto which values are projected.

### Value Atom v

The numerical measurement assigned to each coordinate $(i, j)$. At this level, we only assume a real-valued measurement that can later be transformed (normalized, log-scaled, smoothed, etc.).

### Encoding Atom f(v)

A rule that converts values into visual appearance, typically color.  Conceptually, this atom answers: *how does a change in value become a change in what we see?*

### Ordering Atom P, Q

Permutations of rows and columns that determine where patterns appear in the grid. This atom controls the visual structure: similar rows/columns can be placed next to each other to reveal clusters and blocks.

### Annotation Atom A(row), B(col)

Additional semantic information attached to rows and columns (groups, scores, clusters, labels). This atom enriches the main matrix with contextual meaning without changing the underlying values.

## Physical Constraints

> The laws that any heatmap system must obey

Heatmaps are constrained by **four unavoidable classes of physical limitations**:

### Perceptual Constraints

- Human color discrimination is limited (≈5–12 reliable steps).
- Color perception is non-linear → requires perceptually uniform colormaps.
- Excessive row/column counts easily exceed visual resolution.

### Spatial Constraints

- Screens offer finite 2D space; data may be $n \times m$.
- Each tile has limited area → limited information density.

### Algorithmic Constraints

- Distance computation is inherently $O(n^2)$.
- Seriation / optimal ordering is NP-hard.
- Clustering is sensitive to algorithms and distance definitions.

### Rendering Constraints

- Heatmaps contain thousands to millions of tiles → heavy draw calls.
- SVG output scales poorly with tile count.
- Flattened tables expand to $n \times m$ rows inevitably.

## Zero-Based Optimal Architecture

> If we redesigned heatmaps from scratch, what should they look like?

### Atomic → Modular → Composable

- Each atom (position/value/encoding/order/annotation) must be independent.
- All modules must be pure functions.
- The system must support functional composition.

## Order-first Architecture

Ordering is the structural core. Before any rendering:

- distance must be computed
- ordering must be determined
- clusters may be extracted

Rendering without ordering produces noise instead of structure.

## Rendering-agnostic

- The architecture must be independent of the rendering backend:
    - ggplot2
    - d3.js
    - WebGL
    - GPU tile engines
    - rasterizers

## Layout as a Constraint-solver

A robust layout engine must:

- support annotation panels
- enable adaptive tile sizing
- allow shared row/column order across figures
- compose multi-panel grids cleanly

```pgsql
Data Layer → Model Layer → Encoding Layer → Rendering Layer → Composition Layer
```

This structure ensures long-term scalability and reuse.

## Concept Overview

A heatmap is a **2D encoding scheme** that maps each element of a numerical matrix  $M_{ij}$  to a **colored tile** positioned at $(i, j)$.

It is useful for revealing:

- Patterns and gradients in the data
- Clusters of similar rows or columns
- Structural relationships between variables
- Annotations and metadata aligned with the matrix

Despite its common usage, a heatmap is not simply a colored table—it is the composition of **four independent transformations**:

> **Matrix → Ordering → Encoded Cells → Composed View**

Understanding these layers allows the heatmap to be reusable, modular, and scalable in any technical domain.

## Data Model

A robust heatmap system requires **four formal data structures**:

---

### Matrix Model

Let \(M \in \mathbb{R}^{n \times m} \) be the real-valued matrix to be encoded.

- Rows: \(R = \{r_1, \dots, r_n\}\)
- Columns: \(C = \{c_1, \dots, c_m\}\)
- Each entry \(M_{ij}\) denotes the measurement at row \(r_i\), column \(c_j\).

Typical transformations applied to \(M\) include:

- normalization / scaling
- log or nonlinear transforms
- smoothing or thresholding
- imputation of missing values

The matrix model defines **what data** will be shown.

---

### Ordering Model

A row ordering is a permutation \(P\) over \(R\);  a column ordering is a permutation \(Q\) over \(C\).

Orderings may be derived from:

- hierarchical clustering
- seriation
- spectral ordering
- PCA-based ordering
- domain-defined custom rules

The ordering model defines **where** the data appear visually.

---

### Annotation Model

Optional metadata for rows and columns:

- \(A(r_i)\): attributes associated with row \(r_i\)  
- \(B(c_j)\): attributes associated with column \(c_j\)

Attributes can be:

- categorical (groups, types, clusters)  
- numerical (scores, quality metrics, weights)  

Annotation metadata define **what auxiliary information** accompanies the main matrix.

---

### Encoding Model

A mapping from numerical values to visual appearance, typically color:

$$
f: \mathbb{R} \rightarrow \text{Color}
$$


Types of color encodings:

- continuous sequential
- diverging (with a central midpoint)
- categorical (for annotation strips)
- perceptually uniform maps (recommended)

The encoding model defines **how** values are visually represented.

## Algorithmic Pipeline

The heatmap construction process can be decomposed into **four explicit algorithmic stages**. Each stage is replaceable and independent.

---

### Step A — Transform (Optional)

Purpose: prepare or enhance the matrix.

Operations include:

- normalization
- z-scoring per row or per column
- outlier handling
- thresholding
- smoothing
- dimensionality reduction (optional)

This step creates a transformed matrix ( M' ).

---

### Step B — Reordering (Structural Layer)

Purpose: determine meaningful row and column orderings.

#### 1. Compute a dissimilarity structure

Examples:

- Euclidean distance
- correlation distance ( 1 - r )
- cosine distance
- custom metric

#### 2. Apply an ordering algorithm

Examples:

- hierarchical clustering
- optimal leaf ordering
- spectral seriation
- sorting by first principal component

#### 3. Produce

- row_order
- col_order
- optional cluster labels

This layer defines the **visual structure and pattern revelation**.

---

### Step C — Long Table Construction (Data Layer)

Graphical engines typically operate on a **flattened long format**, not the raw matrix.

For each matrix cell $M_{ij}$, produce a table:

| RowId | ColId | Value |
| ----- | ----- | ----- |
| r₁    | c₁    | v₁₁   |
| r₁    | c₂    | v₁₂   |
| ...   | ...   | ...   |

Then:

1. Convert RowId & ColId to **ordered factors** using the ordering model
2. Join row_meta and col_meta
3. Add any computed attributes (cluster id, normalized score, etc.)

This table is the **universal interface** between data and rendering.

---

### Step D — Rendering (Tile Engine + Layout Composition)

Rendering consists of **two parts**:

#### 1. Tile Engine

- Draw a grid of tiles using (RowId, ColId) as coordinates
- Apply continuous color encoding
- Use a fixed aspect ratio for square cells
- Render legends, labels, and axis guides

#### 2. Layout Composition

- Add annotation panels (top, left, right, bottom)
- Arrange multiple heatmaps in a grid
- Control spacing, proportional sizing, and alignment
- Export as raster (PNG) or vector (SVG/PDF)

Rendering produces the final visual output.

## Design Principles

To ensure **interpretability and technical correctness**:

### Principle 1 — Ordering must reveal structure

**Avoid** alphabetical or arbitrary ordering unless intentional.

### Principle 2 — Color scale must be perceptually monotonic

Use **perceptually uniform scales**; **avoid** misleading gradients.

### Principle 3 — Respect aspect ratio

Square cells **maximize perceptual stability**.

### Principle 4 — Separate absolute vs relative patterns

Row-wise Z-score improves pattern visibility but hides absolute magnitude.

### Principle 5 — Annotation should complement, not dominate

Metadata aids interpretation but should **not overshadow the matrix**.

### Principle 6 — Keep the cognitive load balanced

Too many rows/columns reduce readability; consider grouping or sampling.

## Common Pitfalls

- Using correlation heatmaps without handling **missing data**.
- Cluster dominance by **outliers**.
- Diverging color maps with **incorrect midpoint**.
- **Inconsistent factor** ordering causing misalignment.
- Using **alphabetical sample order** instead of structural order.
- Rendering large heatmaps as SVG → **excessive file size**.
- Blindly using Z-score when **absolute differences matter**.

Being aware of these pitfalls prevents misinterpretation.

---

## Scalability and Performance

Heatmaps become **computationally expensive with large matrices**.

### Bottlenecks

- Distance computation: $O(n^2)$
- Long table expansion: $n \times m$ rows
- Rendering density: every tile is a draw call
- Memory pressure: dense matrix + annotations

### Strategies

- Subsampling or aggregation
- Block-wise clustering or approximate nearest neighbors
- Pre-rendering tiles into raster layers
- Chunked processing for annotation joins
- Conditional rendering based on visible area

A production-level system must account for these constraints.

## Reusability and Software Architecture

A reusable heatmap framework follows **Separation of Concerns**:

### Layer 1 — Data Layer

Matrix + metadata + validation

### Layer 2 — Model Layer

Distance functions
Clustering algorithms
Ordering extraction

### Layer 3 — Encoding Layer

Continuous color mapping
Categorical annotation encoding
Legend generation

### Layer 4 — Rendering Layer

Components:

- Tile engine
- Annotation strips
- Layout engine

Reusability emerges when:

- Each transformation is a pure function
- The pipeline is composable
- Ordering, encoding, metadata, and rendering are decoupled
- Plots share consistent ordering and annotation logic

### Example abstraction

```R
heatmap(matrix) =
  compose(
    render_tiles(
      encode_values(
        reorder(matrix)
      )
    ),
    render_annotations(...)
  )
```

This architecture allows partial reuse (e.g., reuse ordering across figures) or extension (e.g., correlation heatmap, multi-panel heatmap).

## Conclusion & Future Extensions

A heatmap is more than a visualization—it is a **principled system of perceptual encoding**, rooted in cognitive science, information theory, and computational constraints.

Future extensions include:

- GPU-accelerated tile rendering
- Hierarchical / multiscale heatmaps
- Streaming heatmaps for real-time data
- WebGL-based billion-tile renderers
- Adaptive ordering based on interactive filtering
- Embedding heatmaps into automated model diagnostics

Heatmaps will remain a foundational analytical primitive, but their architecture can continue evolving toward more scalable, modular, and perceptually optimized forms.
